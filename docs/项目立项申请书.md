## 一、项目背景与目标

### 1. 项目背景

在数字化市场竞争日趋激烈的商业环境下，企业需要持续关注竞品价格变化、新品发布和负面舆情等外部信号。传统竞品分析依赖人工检索、整理和判断，存在效率低、更新慢、信息分散、结论难以溯源等问题。

随着 Claude 大模型、LangChain、RAG 和 Multi-Agent 技术的发展，可以将公开舆情采集、知识库检索、竞品动态提取、结构化对标和简报生成整合为一套自动化分析流程。本项目以企业实训为背景，构建一个可运行、可展示、可追踪的竞品情报智能分析系统。

### 2. 项目目标

本项目计划实现以下目标：

1. 构建竞品主体和监控维度配置能力，支持价格战、新品发布、负面舆情三类核心维度。
2. 支持公开舆情数据采集、清洗、去重、结构化解析和来源留存。
3. 使用 RAG 构建竞品情报知识库，保证分析结论能够回溯到原始证据。
4. 构建价格监控 Agent、新品检测 Agent、负面舆情 Agent，实现多维度竞品动态分析。
5. 基于 Claude Skill 封装竞品情报分析能力，输出结构化情报卡片和竞争态势简报。
6. 使用 FastAPI 提供服务化接口，便于前端看板、报告导出和后续系统集成。

## 二、监控维度与竞品列表

### 1. 监控维度

1. **价格战**：监控竞品价格调整、促销活动、套餐变化和限时优惠，识别可能影响本企业市场策略的价格信号。
2. **新品发布**：监控竞品新产品上线、功能迭代、版本更新和产品线调整，分析产品竞争压力。
3. **负面舆情**：监控竞品投诉、负面媒体报道、用户集中反馈和服务风险，识别潜在机会点与威胁点。

### 2. 竞品主体列表

初始阶段采用可替换的示例竞品主体，后续可根据实际行业补充：

- 竞品A：目标行业主要竞争企业或产品。
- 竞品B：同价位或同功能定位的竞争企业或产品。
- 竞品C：具备近期新品发布或舆情波动的竞争企业或产品。

每个竞品主体需记录名称、别名、官网地址、行业分类、监控优先级、关注维度和数据源。

## 三、技术选型与架构

### 1. 技术选型

- 大模型：Anthropic Claude 或 Anthropic 兼容接口。
- 开发框架：LangChain。
- 能力封装：Claude Skill。
- 智能体机制：LangChain Agent / Tool Use。
- 检索增强：RAG。
- 向量数据库：Chroma / FAISS。
- 后端服务：FastAPI。
- 开发语言：Python 3.11。
- 配置管理：`.env` + `python-dotenv`。
- 观测与调试：LangSmith 或本地日志。
- 部署方式：本地运行，后续支持 Docker 容器化。

### 2. 五层架构

1. **大模型基础层**：负责 Claude 模型调用、模型参数配置、错误重试和 Prompt 模板组织。
2. **数据处理与知识检索层**：负责公开舆情采集、文本清洗、分块、向量化和 RAG 检索。
3. **智能体与工具层**：负责价格监控 Agent、新品检测 Agent、负面舆情 Agent 的分析任务。
4. **服务化接口层**：使用 FastAPI 封装 RESTful API，支持前端和外部系统调用。
5. **展示与交付层**：提供竞品情报看板、风险预警、Agent 推理追踪和竞争态势简报。

## 五、工程规范

1. 目录结构统一为 `main.py`、`modules/`、`data/`、`chroma_db/`、`docs/`、`README.md`。
2. API Key、模型地址、数据库路径等敏感配置统一放入 `.env`，禁止硬编码。
3. Prompt 模板统一放入 `modules/prompts.py`，便于版本管理和调试。
4. 原始舆情数据必须保留来源、采集时间和原始文本，分析结论必须可溯源。
5. 代码遵循基础 PEP8 规范，核心模块添加必要注释和异常处理。
6. 每日提交程序员日志，记录完成内容、问题、解决方案和次日计划。
